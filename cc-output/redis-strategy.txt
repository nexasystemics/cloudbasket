=== CRITICAL PROBLEM ===

Vercel Serverless functions are stateless. Each cold start gets a FRESH process.
The in-memory Map caches are LOST on every cold start and are NOT shared between
concurrent instances. Under real load:
  - Amazon PA-API: up to 10+ concurrent cold calls per minute
  - Flipkart API: same pattern
  - Exchange rates: re-fetched on every cold start

Net effect: external API rate limits are hit faster than expected, and Vercel
function compute cost is higher than necessary.

=== RECOMMENDED IMPLEMENTATION PATTERN ===

The pattern to apply to each heavy route (example: /api/currency/rates):

  import { getCache, setCache } from '@/lib/redis'

  export async function GET(request: NextRequest) {
    // 1. rateLimit (already in place)
    // 2. Zod validation (already in place)

    const cacheKey = 'currency:INR'
    const cached = await getCache<ExchangeRates>(cacheKey)
    if (cached) return NextResponse.json(cached)

    const data = await getExchangeRates()
    await setCache(cacheKey, data, 3600)   // 3rd arg = TTL in seconds
    return NextResponse.json(data)
  }

For routes with dynamic keys (search queries):

=== CACHE KEY CONVENTIONS (recommended) ===

Prefix convention:
  cb:search:amazon:{q}:{cat}      15 min
  cb:search:flipkart:{q}:{cat}    20 min
  cb:search:cj:{q}                20 min
  cb:deals:flipkart               10 min
  cb:product:amazon:{asin}        30 min
  cb:prices:{productIds_md5}      15 min
  cb:currency:INR                  1 hour
  cb:comp:{name}:{brand}          30 min

All keys use lowercase, colon-separated hierarchy.
Upstash prefix in lib/redis.ts is already 'cb:rl' for rate limiters.
Use 'cb:cache' prefix for application cache to avoid collision.
